# 推理中心 Ollama 生命周期与运行实例设计

## 背景

当前推理中心已经具备基础的：

- Ollama 运行状态展示
- 运行面板
- 已加载模型列表

但整体仍然偏“状态查看”，不符合产品上对推理中心的预期。主要问题是：

- 顶部 Ollama 区域只能查看状态，不能完整管理生命周期
- 运行面板在视觉和交互上仍缺少必填校验与智能辅助入口
- “运行中模型”只是已加载模型列表，还不是可管理的“运行实例”视图

本轮要把推理中心重新定义为：

1. 顶部管理 Ollama 服务生命周期
2. 中部配置并启动模型实例
3. 底部管理 Ollama 管理的多个“已启动模型”实例

## 官方能力边界澄清

本设计基于当前 Ollama 官方公开能力：

- 单个 Ollama 服务可以同时加载多个模型并并发处理请求
- Ollama 官方 API 可返回当前已加载模型列表（`/api/ps`）
- Ollama 服务端口默认是 `127.0.0.1:11434`，可通过环境配置变更服务地址/端口

但 Ollama 官方 API 本身并没有直接提供产品级“实例资源”的完整语义，例如：

- 异常实例
- 配置编辑后自动重启
- 每个实例的管理对话框

因此本产品需要在 UI/业务层定义一个“运行实例”抽象：

- 一个实例 = 一个由 Ollama 管理、当前已启动的模型
- 列表主数据来源仍然基于当前 Ollama 可观察状态
- 产品层补充管理语义、详情视图和编辑入口

本轮只定义产品结构与 UI 行为，不预设超出当前能力边界的后端实现细节。

## 总体信息架构

推理中心页面调整为三段结构：

### 1. 顶部：Ollama 进程状态条

顶部不再使用大卡片，而改为一条压缩状态条，只占据主页面一行。

状态条包含：

- 安装状态
- 运行状态
- 版本号
- 当前服务端口
- 当前运行实例数

右侧操作区包含：

- `管理` 按钮
- 展开/更多入口

点击后弹出 `Ollama 生命周期管理对话框`。

### 2. 中部：运行面板

运行面板继续保留为页面核心操作区，定位是：

- 选择本地已下载模型
- 配置本次启动参数
- 启动一个新的运行实例

本轮补充：

- 必填字段增加 `*`
- 必填校验明确可见
- 面板增加 `智能填写参数` 按钮

说明：

- `智能填写参数` 本轮只做 UI 入口，不实现智能计算逻辑
- 所有 Ollama 字段名继续保持官方原样
- 当前已实现的高密度控制台式排版继续保留

### 3. 底部：运行实例列表

底部区域从“运行中模型”升级为“运行实例”列表。

列表中一行表示一个运行实例。

每行必须至少可见：

- 状态
- 模型名称
- 端口号
- 右侧操作按钮

列表支持两类实例状态：

- 正常运行实例
- 异常实例

异常实例仍必须保留在列表中，以便用户查看详情和处理。

## 顶部状态条设计

### 目标

让主页面顶部只展示摘要，不把服务管理操作直接摊平在页面上。

### 展示内容

建议状态条从左到右展示：

- 状态点 + `Ollama 进程`
- `已安装 / 未安装`
- `运行中 / 未启动 / 安装中 / 异常`
- `版本 x.x.x`
- `端口 11434`
- `实例 N`

### 操作入口

右侧只保留轻量入口：

- `管理`
- `刷新`

不要在状态条上直接铺满“安装 / 卸载 / 停止 / 启动 / 配置”等按钮，避免顶部过重。

## Ollama 生命周期管理对话框

### 目标

把 Ollama 服务的完整生命周期管理从主页面剥离到对话框中。

### 管理范围

对话框内承载声明式管理能力：

- 安装
- 卸载
- 启动
- 停止
- 配置

### 交互要求

对话框内建议分成两个区域：

#### A. 当前服务状态摘要

显示：

- 安装路径
- 版本
- 服务地址 / 端口
- 当前实例数
- 最近状态

#### B. 生命周期操作区

以操作分组形式展示：

- `安装 / 卸载`
- `启动 / 停止`
- `配置`

配置区本轮先作为声明式配置入口，不要求一次性完成完整实现。

### 设计原则

- 主页面只负责摘要展示
- 重操作全部进对话框
- 生命周期动作与运行实例动作分开，不混在主列表中

## 运行面板设计补充

### 必填项

运行面板中的必填参数需要显式增强：

- `model` 增加 `*`
- 所有当前定义为必填的字段在 label 上显示星号
- 校验失败时要有明确可见反馈，不只依赖按钮 disabled 状态

### 智能填写参数按钮

面板新增小按钮：

- `智能填写参数`

放置原则：

- 靠近常用参数区或主操作区
- 视觉层级低于主启动按钮
- 明确是辅助工具，不压过手动配置

本轮只做 UI：

- 按钮可见
- 可点击
- 可以弹出提示、占位 toast 或后续对话框入口

本轮不做：

- 根据本机配置自动计算参数
- GPU 检测联动计算
- 自动落表逻辑

### 不变约束

- 参数字段集合不变
- 字段名不变
- 提交 payload 不变
- 启动流程语义不变

## 运行实例列表设计

### 实例定义

本设计中的“实例”定义为：

> 一个由 Ollama 管理、当前已启动的模型。

这不是“系统任意 Ollama 进程”，也不是“任意历史启动记录”，而是推理中心当前管理视图中的逻辑实例。

### 列表展示字段

每行建议至少包含：

- 状态标识
- 模型名称
- 端口号
- 启动参数摘要
- 操作按钮

其中：

- 状态应可一眼区分 `运行中` 与 `异常`
- 模型名为主信息
- 端口号需固定可见
- 参数摘要只展示少量关键项，不在行内展开所有配置

### 端口号原则

这一点需要明确：

- 从 Ollama 官方能力看，多个已加载模型通常共享同一个服务端口
- 因此本版 UI 所谓“实例端口”，默认显示的是当前服务访问端口
- 不在没有真实后端支持的前提下伪造“一模型一独立端口”的概念

如果未来做真正的多服务实例编排，再扩展为独立实例端口模型。

## 运行实例详情对话框

### 目标

避免在列表主行中塞入过多信息，把详情、参数查看和编辑操作集中到对话框内。

### 对话框内容

详情对话框中应能查看：

- 实例状态
- 模型名称
- 端口号
- 当前启动参数
- 关键运行信息
- 异常信息（如果实例异常）

### 操作能力

详情对话框中提供：

- 查看更多详情
- 修改配置
- 修改启动参数

当启动参数被修改后：

- 应执行“保存后自动重启实例”的产品语义

### 本轮约束

本轮设计只定义这个交互语义，不要求一次性做完全部后端链路。

即：

- UI 和信息架构可以先设计完整
- 详情字段可先使用当前能力可提供的数据
- “修改后自动重启”的产品行为可先定义为后续实现目标

## 状态模型

### Ollama 服务状态

服务状态至少覆盖：

- 未安装
- 已安装未启动
- 启动中
- 运行中
- 停止中
- 安装中
- 异常

### 运行实例状态

实例状态至少覆盖：

- 运行中
- 启动中
- 重启中
- 已停止
- 异常

如果当前后端能力不足以完整感知所有状态，本轮允许先展示：

- 运行中
- 异常
- 未知

但 UI 结构要为更完整状态预留位置。

## 组件边界建议

建议拆分为以下视图组件：

- `OllamaServiceStatusBar`
- `OllamaLifecycleDialog`
- `OllamaRunPanel`
- `OllamaInstanceList`
- `OllamaInstanceRow`
- `OllamaInstanceDetailDialog`

这样可把：

- 服务生命周期管理
- 新实例启动配置
- 已运行实例管理

三类职责完全分开。

## 本轮实施优先级建议

为避免一次性跨度过大，建议按下面顺序落地：

### 第一优先级

- 顶部改为压缩状态条
- 增加生命周期管理对话框 UI 框架
- 运行面板补齐必填星号与 `智能填写参数` 按钮 UI
- 底部列表重构为运行实例列表行样式

### 第二优先级

- 实例详情对话框 UI
- 异常实例展示位
- 参数摘要与详情结构

### 第三优先级

- 生命周期声明式配置表单
- 修改实例参数后自动重启的完整链路
- 智能填写参数的真实计算逻辑

## 不做事项

本轮设计不要求：

- 一次性完成 Ollama 卸载能力的真实后端实现
- 一次性完成智能填写参数的算法实现
- 在没有真实支持的情况下做“一实例一独立端口”的伪装
- 把所有实例详情都堆到主列表行中

## 验收标准

本轮设计完成后，应满足：

- 顶部 Ollama 区域压缩成单行摘要，而不是主页面大卡片
- Ollama 生命周期管理通过对话框承载，而不是散落在主页面
- 运行面板有明确必填标识和智能填写参数入口
- 底部区域不再是简单“运行中模型”，而是可管理的运行实例列表
- 列表行能直观看到状态、模型名、端口号
- 每个实例都有详情入口，并可在详情中承载参数编辑与后续重启语义
