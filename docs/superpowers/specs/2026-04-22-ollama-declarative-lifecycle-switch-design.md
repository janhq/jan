# 推理中心 Ollama 声明式生命周期开关设计
## 背景

当前推理中心的 Ollama 生命周期交互仍然是过程式的：

- 用户点击“启动”就立即调用一次 `start_ollama`
- 用户点击“停止”就立即调用一次 `stop_ollama`
- 前端把单次命令成功/失败直接映射为 UI 成败

这种设计会把实现细节暴露给用户，并且对时间、时机、进程树、健康检查窗口非常敏感。实际表现就是：

- 启停中间态容易被误判成失败
- 前后端都需要处理大量“过程噪声”
- 用户真正关心的是“最后有没有达到运行/停止的目标状态”，而不是中间每一步是否顺利

因此，本轮把 Ollama 生命周期改成声明式目标状态模型：

> 用户不再直接执行“启动一次”或“停止一次”命令，而是声明系统最终应该处于“运行中”还是“已停止”。

系统的职责是在一个可接受时间窗口内收敛到目标状态。

## 目标

本设计只解决 Ollama 服务级生命周期交互，不涉及：

- 运行参数面板
- 模型实例详情
- 模型加载/卸载逻辑

本轮目标是：

1. 把顶部 Ollama 生命周期交互改成单个滑动开关
2. 用“期望状态”驱动前后端收敛逻辑
3. 消除单次 start/stop 调用直接决定 UI 成败的机制
4. 在调整期间禁止快速反复拨动，优先保证稳定性和可理解性

## 用户交互设计

顶部 Ollama 区域改为一个声明式开关：

- 开关为 `开`：表示“期望 Ollama 最终运行”
- 开关为 `关`：表示“期望 Ollama 最终停止”

用户拨动后：

- UI 不立即按单次命令成功/失败改变结果
- UI 进入“调整中”状态
- 系统开始收敛到目标状态
- 只有在超时后仍未收敛，才显示失败

### 调整中的交互约束

为降低实现复杂度并避免竞态，本轮明确采用：

- 调整期间禁止再次拨动开关
- 不支持“启动中又立刻要求停止”
- 不支持“停止中又立刻要求启动”

因此生命周期稳定态只有两种：

- `已停止`
- `运行中`

过渡态只有两种：

- `启动中`
- `停止中`

## 前端状态模型

前端不再把 Ollama 仅表示为 `isRunning: boolean`，而是引入生命周期视图状态：

- `stopped`
- `starting`
- `running`
- `stopping`
- `error`

同时维护一个单独的期望状态：

- `desiredRunning: boolean`

约束如下：

- `desiredRunning = true` 时，系统应尝试收敛到 `running`
- `desiredRunning = false` 时，系统应尝试收敛到 `stopped`
- `starting/stopping` 期间开关禁用
- `error` 不是目标状态，只是本次收敛失败后的展示态

### UI 映射

- `stopped`：开关关闭，可拨到开
- `starting`：开关视觉上保持开启，但禁用，并显示“正在启动”
- `running`：开关开启，可拨到关
- `stopping`：开关视觉上保持关闭，但禁用，并显示“正在停止”
- `error`：保留最近一次稳定态对应的开关位置，并显示“未达到期望状态”

## 后端能力模型

后端不再只暴露过程式命令给 UI，新增两个幂等的声明式命令：

- `ensure_ollama_running`
- `ensure_ollama_stopped`

它们的语义不是“执行某一步动作”，而是：

- 读取当前状态
- 如有必要执行启动/停止动作
- 在超时时间内轮询健康状态
- 直到达到目标状态或超时

### `ensure_ollama_running` 语义

1. 先检查 Ollama 当前是否已运行
2. 如果已运行，直接返回成功
3. 如果未运行，执行正确的 Windows 启动入口
4. 在超时时间内轮询 `127.0.0.1:11434/api/version`
5. API 可用则返回成功
6. 超时仍不可用则返回失败

### `ensure_ollama_stopped` 语义

1. 先检查 Ollama 当前是否已停止
2. 如果已停止，直接返回成功
3. 如果仍在运行，执行停止收敛逻辑
4. 在超时时间内轮询 `127.0.0.1:11434/api/version`
5. API 不可达则返回成功
6. 超时仍可达则返回失败

## 成败判定原则

这是本设计最核心的变化。

成功/失败不再基于：

- 某一次 `taskkill` 的返回码
- 某一次 `spawn()` 是否成功
- 某一条中间日志是否报错

而只基于：

> 在声明的时间窗口内，系统最终是否达到了期望状态。

因此：

- 中间步骤失败但最终收敛成功：整体算成功
- 中间步骤成功但最终未收敛：整体算失败

过程性错误只进入日志，不直接成为用户层失败判断依据。

## 日志与可观测性

为了支持后续排障，生命周期收敛过程需要补充结构化日志。

### 前端日志

记录：

- 用户声明的目标状态
- 当前页面 URL
- 生命周期状态切换
- 收敛结果（成功/超时失败）

示例目标：

- `Requested Ollama desired state: running`
- `Requested Ollama desired state: stopped`
- `Ollama reconciliation failed: desired=running`

### 后端日志

记录：

- 进入 `ensure_ollama_running/stopped`
- 初始状态判断
- 是否执行了真实 start/stop 动作
- 每轮健康检查结果
- 最终是否在时限内收敛

中间错误保留，但只作为 `meta` 或 warning/error 上报，不直接作为用户交互结果。

## 组件边界建议

建议把顶部服务级生命周期逻辑拆成两个层次：

### 1. 展示组件

- `OllamaServiceStatusBar`

职责：

- 渲染当前状态
- 渲染生命周期开关
- 渲染“调整中/失败”提示

不负责直接调用后端。

### 2. 状态协调 Hook

- 可在 `useOllamaStatus` 内扩展，或抽出 `useOllamaLifecycleController`

职责：

- 管理 `desiredRunning`
- 管理 `starting/stopping/running/stopped/error`
- 调用 `ensure_ollama_running/stopped`
- 统一处理超时、日志和状态刷新

这层是声明式设计的核心，不应散落到页面按钮回调里。

## UI 文案建议

统一从“动作导向”改成“目标状态导向”：

- `启动 Ollama` -> `期望运行`
- `停止 Ollama` -> `期望停止`
- `启动失败` -> `未能在预期时间内进入运行状态`
- `停止失败` -> `未能在预期时间内进入停止状态`
- `正在启动` / `正在停止` -> `正在调整到期望状态`

这样更符合声明式控制模型，也能降低用户对单次命令执行过程的误解。

## 非目标

本轮不做以下事项：

- 调整期间支持反向拨动
- 多个生命周期收敛任务并发
- 生命周期任务队列
- 模型实例级开关与服务级开关联动
- 保存生命周期偏好到持久化配置

这些都属于后续增强项，不进入本轮。

## 验收标准

本轮完成后应满足：

1. 顶部 Ollama 生命周期交互改为单一滑动开关
2. 开关表达的是“期望最终状态”，不是“立即执行一次动作”
3. 调整期间开关禁用，不能快速反复拨动
4. 前端不再直接调用裸 `start_ollama/stop_ollama` 作为最终交互语义
5. 后端提供幂等的目标状态收敛命令
6. 最终成败以是否在超时内达到目标状态为准
7. 启停过程中的中间错误仅写日志，不直接打断用户

## 推荐实现顺序

1. 后端新增 `ensure_ollama_running` / `ensure_ollama_stopped`
2. 前端 Hook 改为生命周期状态机
3. 顶部状态栏 UI 改为 `Switch`
4. 清理旧的按钮式 start/stop 入口
5. 增加前后端日志与测试
